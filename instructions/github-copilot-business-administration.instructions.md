# GitHub Copilot Business & Enterprise Administration

Guidance for helping **organization and enterprise administrators** provision, troubleshoot, and manage GitHub Copilot Business / Copilot Enterprise access. Use this when a user reports that Copilot is "not enabled", seats will not assign, provisioning appears stuck, or the API returns unexpected results for an org.

> **Scope and boundaries**
>
> - This guidance covers **supported, documented** administration only.
> - It never advises bypassing billing, risk, compliance, trust-and-safety, or verification controls. Those controls are operated by GitHub and can only be resolved by GitHub Support or the responsible internal team.
> - If a restriction is a deliberate hold (for example a date-locked upgrade restriction, a payment hold, or a compliance/eligibility review), the **only** correct path is to escalate to GitHub Support with complete evidence — not to work around it.

## Instructions

- **Confirm the plan before debugging.** Copilot Business and Copilot Enterprise are billed at the organization or enterprise level. Verify which plan the org is actually on before assuming a bug. An org still reporting a free/unprovisioned plan via the API has not finished activation, regardless of what the UI attempted.
- **Verify the requester's role.** Seat and policy management requires an **organization owner** (or an enterprise owner / billing manager for enterprise-level settings). Members and billing managers have different scopes.
- **Separate the two layers.** Personal-account Copilot status and organization-level provisioning are independent. A restriction on a personal account does not provision an org, and an org seat does not lift a personal-account restriction. Diagnose each layer separately.
- **Read the actual API/HTTP response.** Status codes are diagnostic:
  - `404` on Copilot org endpoints usually means **Copilot Business is not enabled / not finished provisioning** for that org — not necessarily a permissions error.
  - `403` typically means **insufficient scope or role** (token missing `manage_billing:copilot` / `admin:org`, or the caller is not an owner).
  - `422` usually means the **request is well-formed but rejected** (e.g., seat assignment for a user who is not an org member, or a policy that conflicts with enterprise settings).
- **Distinguish a misconfiguration from a hold.** If local configuration (roles, scopes, policies) checks out but the feature still will not enable, treat it as a backend/account state issue and escalate rather than continuing client-side troubleshooting.

## Common provisioning issues and how to diagnose them

| Symptom | Most likely cause | First check |
|---------|-------------------|-------------|
| `404` from Copilot org endpoints | Copilot Business never finished enabling for the org | Confirm the org's plan in **Settings → Billing**; re-run enablement as an owner |
| Org "plan" still shows free/unprovisioned in API | Activation started in UI but backend did not complete | Compare UI state vs. billing API; if mismatched, escalate |
| Seats will not assign (`422`) | Target users are not org members, or seat policy excludes them | Verify membership and the seat-assignment policy (selected teams/users vs. all members) |
| `403` on management calls | Token scope or caller role insufficient | Confirm owner role + token scopes (`manage_billing:copilot`, `admin:org`) |
| "Upgrade not available until \<date\>" | Automated billing / risk / eligibility hold | **Not self-service.** Escalate to Support — see below |
| Enablement blocked despite valid admin + scopes | Account/org under automated review or hold | **Not self-service.** Escalate to Support — see below |

### About automated holds, date-locks, and eligibility reviews

Some restrictions are produced by automated billing, risk, trust-and-safety, or compliance systems rather than by normal admin settings:

- A hard date-locked message (e.g., *"upgrade not available until \<date\>"*) is generated by an automated system and has **no self-service clear path** in account settings or the API.
- Provisioning of advanced commercial features can be held pending account/eligibility verification.
- These holds are intentional. Do **not** advise the user to rename an org, create a parallel org, switch accounts, or otherwise attempt to evade the control. Advise them to escalate and supply evidence so the responsible team can review and, if appropriate, lift the hold.

## Useful API endpoints (read first, change carefully)

Use the REST API to gather evidence before escalating. Calls require an organization owner and a token with `manage_billing:copilot` (and `admin:org` where noted).

- **Copilot billing for an org**
  `GET /orgs/{org}/copilot/billing`
  Returns seat breakdown and the current Copilot plan. A `404` here is the key signal that Copilot Business is not enabled for the org.
- **List Copilot seats**
  `GET /orgs/{org}/copilot/billing/seats`
- **Assign seats to selected users / teams**
  `POST /orgs/{org}/copilot/billing/selected_users`
  `POST /orgs/{org}/copilot/billing/selected_teams`
- **Remove seats**
  `DELETE /orgs/{org}/copilot/billing/selected_users`

Always capture the **full request (endpoint + method), the response status code, and the response body** when documenting an issue — these are exactly what Support needs.

## Escalating to GitHub Support

When the issue is a backend/account state problem or an automated hold, open a Support ticket. Route it under: **Copilot → Billing / Provisioning → Organization access.**

Include all of the following so the ticket can be actioned without back-and-forth:

1. **Organization slug** (and enterprise slug, if applicable).
2. **Confirmation of org-owner access** for the requester.
3. **Screenshot of any hold or date-lock message** (e.g., the "upgrade not available until \<date\>" notice), with the date clearly visible.
4. **The `404` / provisioning response** from `GET /orgs/{org}/copilot/billing`, including status code and body.
5. **API output showing the org's current plan** (especially if it still reports free/unprovisioned).
6. **A clear, explicit request**, for example: *"Please review Copilot Business provisioning for this organization and confirm whether an account/eligibility verification or compliance hold is blocking activation."*

Also ask Support to confirm, explicitly:

- Whether a **personal-account restriction** and the **organization restriction** are linked.
- Whether the org is subject to an **automated eligibility/compliance review** (and what evidence, if any, would resolve it).
- Whether **backend provisioning can be completed manually** once verification passes.

## Additional Guidelines

- Frame answers around the two-layer model (personal account vs. org provisioning) so the user does not chase the wrong layer.
- Prefer pointing to **Settings → Billing** and the official Copilot billing API over speculative fixes.
- When configuration is already correct, say so plainly and direct the user to escalation rather than inventing further client-side steps.
- Never suggest circumventing a hold; suggest documenting it thoroughly and escalating through the supported channel.
